# Story 4.2: Talent Override Icon Resolution & Log Spam Fix

**Status:** Done
**Epic:** 4 — Diagnostics & v0.3.0
**Version:** v0.3.1
**Commits:** `7ac1a21`

## Context

Blood DK players using Defile (a talent that replaces Death and Decay) saw the Death and Decay icon instead of Defile's icon. `C_AssistedCombat.GetNextCastSpell` returns the **base** spellID (`43265`) even when a talent overrides the spell (`152280`). `C_Spell.GetSpellInfo(43265)` gives Death and Decay's icon, not Defile's. This was confirmed via the `RAW_SUGG` session log entries.

Separately, the `SKIPPED` DLog tag (used to note when the proc-alert spell occupies a queue slot) fired every 50ms poll tick during a proc, generating 50+ identical entries and flooding the 500-entry ring buffer with useless noise.

## Acceptance Criteria

- [x] When the suggested spell is talent-overridden, the displayed icon matches the override (e.g., Defile not Death and Decay)
- [x] The spell name displayed in status/log output also matches the override
- [x] Every talent substitution is recorded in the session log with the `OVERRIDE` tag
- [x] The `SKIPPED` log entry fires once per proc activation, not every poll tick

## Implementation Notes

- Override resolution (see ADR-9): after finding `realSlotID`, call `GetActionInfo(realSlotID)` in a pcall; the bar slot returns the override spellID. If it differs from the base ID, substitute it before rendering
- `lastSkippedAlertID` change-detection guard: the SKIPPED DLog is only written when `currentAlertSpellID` differs from the last-logged value; reset on `ALERT_HIDE` and when the suppress path runs

**Known open bug:** `GetRotationSpells()` returns base IDs; the `sid ~= primaryID` deduplication check in `GetSuggestionQueue` compares base IDs against the override `primaryID` and may miss matches, causing the same ability to appear in both primary and secondary slots.

## Files Changed

- `HekiLight.lua` — override resolution block in `GetActiveSuggestion()`, `lastSkippedAlertID` guard, `OVERRIDE` DLog tag
